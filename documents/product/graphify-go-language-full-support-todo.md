# Graphify-Go 五语言完整支持开发计划 (TODO)

> **目标**: 在不影响现有功能与输出协议的前提下，将 `JavaScript / TypeScript / Python / Go / Verilog` 从“基础支持”升级为“工程可用的完整支持”。
> **范围**: 聚焦五门目标语言，不做其他新语言扩展。

## 0. 完整支持定义 (Definition of Done)

- [ ] **语法覆盖达标**: 每门语言都覆盖声明、调用、依赖三类核心关系，并通过对应语言用例集验证。
- [ ] **输出协议不变**: `graph.json` 字段结构与现有协议保持 100% 兼容。
- [ ] **旧行为不回归**: 同一测试夹具在改造前后，关键节点与边关系不劣化。
- [ ] **错误可降级**: 单文件解析失败不会阻断全量扫描，日志可定位失败原因。

## 1. 基线冻结与回归护栏 (Phase 1)

- [ ] **冻结基线**: 固定一组 JS/TS/Python/Go 夹具，保存当前 `graph.json` 快照作为回归基线。
- [ ] **建立对比脚本**: 对节点数、边数、关系类型分布和关键节点 ID 做自动 diff。
- [ ] **补全现有测试缺口**: 为 TypeScript、Python 增加与 JS/Go 同等级的 `extractor` 测试。
- [ ] **加 CI 门禁**: `go test ./...` + 快照对比必须全绿才允许合入。

## 2. JavaScript 完整支持 (Phase 2 - JS)

- [ ] **声明覆盖**: 补齐函数表达式、箭头函数、对象方法、导出声明等常见定义形态。
- [ ] **调用覆盖**: 补齐 `call/new`、链式调用、可选链调用等主流模式。
- [ ] **依赖覆盖**: 同时覆盖 `import`、`require`、动态导入等依赖入口。
- [ ] **回归验收**: 在 Node 工程夹具上验证提取结果稳定，快照 diff 无异常放大。

## 3. TypeScript 完整支持 (Phase 3 - TS)

- [ ] **TS 专有语法**: 增加 `interface/type alias/enum/namespace` 的提取策略与映射规则。
- [ ] **类与方法增强**: 覆盖访问修饰符、泛型、抽象类与实现关系的关键节点提取。
- [ ] **类型导入与路径别名**: 覆盖 `import type` 与常见路径别名场景。
- [ ] **回归验收**: 在真实 TS 项目夹具中验证无漏提取和无明显误提取。

## 4. Python 完整支持 (Phase 4 - Python)

- [ ] **声明覆盖**: 覆盖嵌套函数、类方法/静态方法、装饰器函数等常见结构。
- [ ] **调用覆盖**: 覆盖属性调用、链式调用、上下文对象方法调用等模式。
- [ ] **依赖覆盖**: 覆盖 `import` 与 `from ... import ...`，并规范模块名提取。
- [ ] **语义一致性**: 明确函数与方法区分策略，避免统计口径漂移。

## 5. Go 完整支持 (Phase 5 - Go)

- [ ] **声明覆盖**: 覆盖函数、方法、接口、结构体及关键类型声明映射。
- [ ] **调用覆盖**: 覆盖选择器调用、包函数调用、常见链路调用模式。
- [ ] **依赖覆盖**: 覆盖标准 `import`、别名导入和多段导入块。
- [ ] **工程验收**: 以中大型 Go 仓库作为夹具，验证性能与准确率不倒退。

## 6. Verilog 完整支持 (Phase 6 - Verilog)

- [ ] **语法覆盖**: 覆盖 `module`、`function/task`、`always`、实例化等主干结构提取。
- [ ] **关系映射**: 定义 Verilog 到统一图模型 (`class/function/method/call/import`) 的稳定映射规则。
- [ ] **依赖覆盖**: 覆盖 `` `include`` 与常见跨文件引用入口，统一模块名与文件名清洗规则。
- [ ] **验收夹具**: 基于真实 RTL 工程夹具验证节点提取率、关系准确率和回归稳定性。

## 7. 通用抽取质量提升 (Phase 7 - Cross-Lang)

- [ ] **去重策略优化**: 统一同名实体合并与标签规范，降低伪重复节点。
- [ ] **噪音控制**: 过滤低价值占位节点，避免污染 God Nodes 与社区分析。
- [ ] **错误观测性**: 为解析失败增加可聚合的错误统计，不中断主流程。
- [ ] **性能守护**: 对大仓库执行时间与内存占用设置回归阈值。

## 8. 文档与发布 (Phase 8)

- [ ] **README 对齐**: 将“已支持语言”从基础支持升级为明确的覆盖声明与边界声明。
- [ ] **PRD 对齐**: 更新 `graphify-go-PRD` 中的多语言能力描述与验收口径。
- [ ] **变更记录**: 输出每个阶段的“新增覆盖点 + 已知限制 + 回归结果”。
- [ ] **发布准入**: 满足测试全绿、快照稳定、性能阈值通过后再发版。

## 9. 不可妥协红线 (Red Lines)

- [ ] **不改协议**: 不变更 `graph.json` 对外协议字段。
- [ ] **不破坏旧能力**: 任何单语言增强不得导致其他语言质量下降。
- [ ] **不牺牲稳定性**: 解析失败必须可降级，不得让 CLI 整体退出。
- [ ] **不做范围蔓延**: 本计划仅限 `JavaScript / TypeScript / Python / Go / Verilog` 五门语言，不新增其他语言。

## 10. 里程碑与责任人 (Milestones)

- [ ] **M1 - 基线与护栏** (1 周): 完成阶段 1，建立回归守护能力。
- [ ] **M2 - JS/TS 完整支持** (2~3 周): 完成阶段 2~3 并通过回归。
- [ ] **M3 - Python/Go 完整支持** (2~3 周): 完成阶段 4~5 并通过回归。
- [ ] **M4 - Verilog 完整支持** (2 周): 完成阶段 6 并通过 RTL 夹具回归。
- [ ] **M5 - 收敛发布** (1 周): 完成阶段 7~8，满足发版准入条件。
